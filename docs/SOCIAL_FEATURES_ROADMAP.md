# 플리로그 — 소셜 기능 로드맵 (추후 추가 기능)

> 작성: 2026-06-21 · 대상 앱: 플리로그 / plilog
> 목적: 현재 "혼자 녹음 → 보관" 도구에 **소셜 레이어**를 얹어, 친구들과 노래로 노는
> 리추얼을 만든다. BeReal·Setlog의 바이럴 메커니즘을 "목소리/노래"로 가져온다.
> 상태: **브레인스토밍 합의본.** 즉시 구현이 아니라 추후 단계별로 추가할 기능 정의서.

---

## 0. 한눈에 보는 방향

| 항목 | 내용 |
|---|---|
| **컨셉 한 줄** | "친구들과 한 소절씩 이어부르는, 목소리로 하는 단톡방" |
| **차별점** | BeReal = 사진, Setlog = 영상. **플리로그 = 목소리/노래** (미점령 영역) |
| **기존 자산 유지** | 개인 녹음 보관함 = **"내 연습실"**로 그대로 존속 |
| **새 헤드라인** | 그 위에 **"친구 방"** (피드 + 릴레이)을 얹는다 |
| **곡 데이터** | 폴더 등록 방식 → **iTunes Search API**로 교체 |

소셜 레이어는 **선택적 부가 모드**다. 혼자 쓰는 사용자는 지금처럼 연습실만 써도 된다.

## 1. 기능 A — 곡 검색/추천 (iTunes Search API)

### 목적
현재 "내가 등록한 폴더에서 선택"하는 곡 지정 방식을, 실제 곡 메타데이터 검색으로 교체한다.

### 동작
- **곡 검색:** 제목/가수로 검색 → 제목·가수·앨범아트·30초 미리듣기 반환
- **녹음에 곡 태그:** 내 녹음(Version)에 "무슨 곡인지" 메타데이터를 붙임
- **오늘의 추천곡 (하루 1회 푸시):** 매일 1회 추천곡을 푸시로 전달.
  원곡 30초 미리듣기를 들려주고 → 그 분위기로 릴레이/업로드를 유도하는 **마중물**.
  - (랜덤 타이밍 알림이 아니라 정기 1회. 노래는 소리를 내야 해서 BeReal식 "지금 당장"이 부적합.)

### 구현 메모
- **API:** [iTunes Search API](https://developer.apple.com/library/archive/documentation/AudioVideo/Conceptual/iTuneSearchAPI/)
  - 인증/키 **불필요**, 무료, K-pop 커버리지 양호
  - 예: `https://itunes.apple.com/search?term=아이유&country=kr&media=music`
  - 반환: `trackName`, `artistName`, `artworkUrl100`, `previewUrl`(30초 m4a)
- **노래방 번호(TJ/금영)는 불필요** — 우리 앱은 노래방에 가는 게 아니라 직접 부르므로
  곡 제목/가수/앨범아트만 있으면 충분. (TJ/금영 공식 무료 API도 없음 → 비공식 스크래퍼뿐이라 배제.)
- **대안 API:** Deezer(키 불필요·30초 프리뷰), Spotify(추천 데이터 풍부하나 OAuth 필요)
- **백엔드 불필요:** 이 기능은 클라이언트에서 직접 호출 가능 → **소셜 기능보다 먼저, 단독으로 출시 가능.**

### 데이터 모델 영향
```typescript
// Version에 곡 메타데이터(선택) 추가
interface Version {
  // ...기존 필드
  track?: {
    trackName: string;
    artistName: string;
    artworkUrl?: string;
    previewUrl?: string;   // 30초 미리듣기
    itunesTrackId?: number;
  };
}
```

## 2. 기능 C — 녹음 트리밍 (로컬 편집)

> 소셜 기능이 아니라 **연습실(보관함) 자체의 품질 개선**이다. 기능 A처럼 서버 없이
> 단독 출시 가능 → Phase 1 묶음에 같이 둔다.

### 목적
보관함의 녹음(Version)에서 앞/뒤 군더더기(시작 전 침묵, 잘못 시작한 부분, 끝 잡음)를
잘라낸다. 순수 로컬 편집이라 백엔드 불필요.

### 동작
- 버전 상세에서 **[편집]** → 파형 위에서 **시작/끝 핸들 드래그**로 남길 구간 지정
- 지정 구간 **미리듣기** → 저장
- 저장 시 **새 버전으로 저장** / **원본 덮어쓰기** 중 선택
  - 덮어쓰기는 되돌릴 수 없으므로 저장 전 경고
- **범위 밖(추후 확장 여지):** 중간 구간 삭제·분할, 여러 녹음 합치기, 페이드 인/아웃,
  볼륨 조절. 1차는 **앞/뒤 트림만**.

### 구현 메모 (라이브러리 탐색 결과)
- `expo-audio`에는 편집 API 없음(녹음/재생 전용) → 별도 수단 필요.
- `ffmpeg-kit-react-native`는 **2025년 초 공식 리테이어드(아카이브)** → 신규 채택 부적합.
- **권장: 네이티브 무손실 트림 모듈을 Expo config plugin으로 래핑**
  - iOS: `AVAssetExportSession` (passthrough preset)
  - Android: `MediaExtractor` + `MediaMuxer`
  - m4a/AAC 컨테이너를 **재인코딩 없이** 구간 복사 → 빠르고 음질 손실 없음.
    컷 단위는 AAC 프레임(~21ms) 경계라 체감 문제 없음.
  - **Expo Go 불가, dev build 필요** (이미 EAS 설정 존재).
- 파형 UI: 진폭 샘플을 추출하거나 간단히 샘플링해서 그림.
- **백엔드 불필요:** 클라이언트 + 네이티브 모듈만으로 완결 → 소셜 기능보다 먼저 출시 가능.

### 데이터 모델 영향
```typescript
// Version에 편집 출처/구간 정보(선택) 추가
interface Version {
  // ...기존 필드
  editedFrom?: string;                     // 원본 versionId (새 버전 저장 시 출처 추적)
  trim?: { start: number; end: number };   // 적용된 구간(초) — 디버깅/되돌리기용(선택)
}
```

## 3. 기능 B — 친구 방 (소셜의 그릇)

### 목적
친구들끼리 방을 만들고, 그 안에서 노래로 소통한다. **피드(수동)**와 **릴레이(능동)**가
**한 방 안의 두 탭**으로 공존한다.

### 동작
- 친구를 초대해 방 생성 (초대 링크/코드)
- 방 안 탭: **[피드]** · **[릴레이]**
- 멤버십, 알림 설정은 방 단위

### 데이터 모델 (신규)
```typescript
interface Room {
  id: string;
  name: string;
  ownerId: string;
  inviteCode: string;
  createdAt: Date;
}
interface Membership {
  roomId: string;
  userId: string;
  joinedAt: Date;
}
```

## 4. 기능 B-1 — 피드 (셋로그/인스타형 · 수동 소셜)

### 목적
부담 없이 "내가 부른 노래"를 방에 올리고, 친구들이 보고 댓글로 반응한다. 릴레이보다 진입장벽이 낮은 **일상 공유** 레이어.

### 동작
- 내 녹음(또는 즉석 녹음)을 방 피드에 **포스트**
- 친구들이 피드에서 재생 → **댓글 / 리액션(ㅋㅋ, 박수, 하트 등)**
- 인스타처럼 시간순 카드 피드. 곡 태그(기능 A)가 붙어 있으면 앨범아트와 함께 노출

### 데이터 모델 (신규)
```typescript
interface Post {
  id: string;
  roomId: string;
  authorId: string;
  versionId: string;     // 올린 녹음
  caption?: string;
  createdAt: Date;
}
interface Comment {
  id: string;
  postId: string;
  authorId: string;
  text: string;
  createdAt: Date;
}
interface Reaction {
  targetId: string;      // postId 또는 relaySegmentId
  userId: string;
  emoji: string;
}
```

## 5. 기능 B-2 — 이어부르기 릴레이 (능동 협업 · 킬링 포인트)

### 목적
친구들이 **한 소절씩 즉흥으로 이어불러** 한 곡을 함께 완성한다. 턴제 FOMO + 즉흥의 의외성 +
완성된 웃긴 결과물이라는 3중 보상. **이 앱만의 차별 메커니즘.**

### 핵심 루프
```
누군가 릴레이 시작 (아무 노래나 한 소절, 아카펠라)
   ↓
다음 사람에게 "🎤 네 차례야!" 푸시
   ↓
앞 조각 듣기 → 분위기 봐서 즉흥으로 이어부르기 (15~30초)
   ↓
다음 사람에게 넘어감 … (계속)
   ↓
마감(수동) 또는 24h 무응답(자동) → 친구 목소리가 섞인 한 곡 완성 → 방 피드에 카드로
   ↓
서로 리액션
```

### 확정된 설계 규칙
- **곡:** 정해진 곡 없는 **즉흥/프리스타일.** 가사/곡 데이터 불필요.
- **접착제:** **반주 없는 아카펠라.** 다음 사람은 앞 조각을 듣고 바로 이어부름.
  (음정/박자가 어긋나는 게 오히려 웃긴 포인트 — 정돈된 결과물을 목표하지 않는다.)
- **타이밍:** **상시 턴제.** 내 차례가 오면 "네 차례" 푸시, 원할 때 부름. 노래방 단톡방 느낌.
- **턴 순서:** **열린 차례** — 방 안 누구나 다음 조각을 집을 수 있되 **연속 2번 금지.**
  한 명이 잠수해도 줄이 안 막힌다. "네 차례" 추천 푸시는 가장 오래 안 부른 사람에게.
- **끝맺음:** 한 방에 **여러 릴레이 스레드 동시 진행.** 각 스레드는 누가 마감하거나
  24시간 조용하면 자동 완성본이 됨. 완성본은 방 피드에 카드로 쌓임.

### 데이터 모델 (신규)
```typescript
interface RelayThread {
  id: string;
  roomId: string;
  starterId: string;
  status: 'open' | 'sealed';
  createdAt: Date;
  sealedAt?: Date;
}
interface RelaySegment {
  id: string;
  threadId: string;
  authorId: string;
  order: number;         // 이어붙인 순서
  storageUrl: string;    // 아카펠라 조각 오디오
  duration?: number;
  createdAt: Date;
}
// 완성본은 RelaySegment를 order 순서로 이어붙여 재생/공유
```

## 6. ⚠️ 가장 큰 결정사항 — 백엔드가 필요하다

현재 플리로그는 **100% 로컬**이다 (AsyncStorage + expo-file-system, 서버 없음).
이 문서의 **모든 소셜 기능(방·피드·댓글·릴레이·푸시)**은 본질적으로 서버를 요구한다:

| 필요 요소 | 이유 |
|---|---|
| **계정/인증** | 친구를 식별하고 방 멤버를 관리하려면 |
| **서버 DB** | Room/Post/Comment/RelaySegment를 기기 간 공유하려면 |
| **오디오 호스팅** | 녹음 파일을 친구가 받아 재생하려면 (로컬 파일은 공유 불가) |
| **푸시 서버** | "네 차례야", "오늘의 추천곡" 알림 (expo-notifications + 서버 트리거) |
| **실시간 동기화** | 릴레이 턴/피드 갱신을 즉시 반영하려면 |

> 예전 Firebase → 로컬 마이그레이션을 **일부 되돌리는** 셈이다.
> **권장 스택 후보:** Supabase(Postgres + Auth + Storage + Realtime, 무료 티어 넉넉) /
> Firebase 복귀 / 자체 백엔드. → **별도 설계 필요.**

**기능 A(곡 검색/추천)와 기능 C(녹음 트리밍)는 백엔드 없이 단독 출시 가능**하다는 점이 중요하다.

## 7. 단계별 출시 로드맵

| 단계 | 범위 | 백엔드 |
|---|---|---|
| **Phase 0 (현재)** | 개인 녹음 보관함 (연습실) | 로컬 |
| **Phase 1** | **기능 A** — iTunes 곡 검색 + 곡 태그 + 오늘의 추천곡(로컬 알림) · **기능 C** — 녹음 트리밍 | 불필요 |
| **Phase 2** | 백엔드 도입 + **기능 B / B-1** — 친구 방 + 피드 + 댓글 (수동 소셜) | 필요 |
| **Phase 3** | **기능 B-2** — 이어부르기 릴레이 (킬링 포인트) | 필요 |

릴레이(Phase 3)를 마지막에 두는 이유: 방·멤버십·오디오 호스팅·푸시 인프라가
Phase 2에서 먼저 갖춰져야 릴레이를 얹을 수 있기 때문.

## 8. 열린 질문 (추후 결정)

- 백엔드 스택 확정 (Supabase 유력?)
- 계정 방식 (소셜 로그인? 전화번호? 익명+코드?)
- 오디오 조각 길이 상한 / 릴레이 완성본 최대 길이
- 피드/릴레이 콘텐츠의 휘발성 (영구 보관 vs 24h 후 사라짐 — BeReal식 휘발 도입 여부)
- 방 인원 상한
- 신고/차단 등 모더레이션 (공개 공유는 아니지만 친구 방이라도 최소한 필요)
- 수익화 방향 (있다면)
- (기능 C) 무손실 트림의 컷 정밀도 — AAC 프레임 경계(~21ms)면 충분한가, 더 정밀하면 재인코딩 필요
- (기능 C) 원본 덮어쓰기 시 복구 불가 경고 UX / 기본값을 "새 버전 저장"으로 둘지
```
